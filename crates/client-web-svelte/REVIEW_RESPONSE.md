# Review Response — Lint/Refactor Sweep

Response to a critical/devil's-advocate review of this PR's changes
(bug fixes, lint sweep, complexity reduction, CSS scoping, color tokens).
The review was generated by a subagent asked to find real problems, not
rubber-stamp. Each finding is evaluated below — accepted, rejected, or
noted for discussion — with the concrete resolution.

---

## Finding #1 — BLOCKER · ACCEPTED · FIXED
**`m` and `f` player shortcuts dropped `preventDefault()`**
(`playerShortcuts.ts:122-124`)

The reviewer is correct. My `KEY_HANDLERS` map omitted the `ev` parameter
on the `m` and `f` handlers, so `event.preventDefault()` was never called
for mute/fullscreen. The original `switch` called it on every handled key.
This is a real regression — `f` would trigger browser fullscreen *and*
the player's handler (double-toggle risk), and `m` is harmless on most
browsers but the contract was "player owns these keys."

**Resolution:** Restored `ev.preventDefault()` on `m` and `f`. Kept
`Escape` without it (original behavior — Escape is a browser-reserved
key and preventDefault on it is a no-op anyway).

---

## Finding #2 — BLOCKER · ACCEPTED · FIXED
**`.player-primary-button` loses cascade order battle**
(`player.css` vs `app.css:1840-1844`)

The reviewer caught a real CSS ordering bug. `.player-icon-button` and
`.player-primary-button` are equal-specificity single-class selectors;
the primary button only overrides the icon button by being declared
*later* in source order. My refactor moved `.player-icon-button` into
the component-imported `player.css` while leaving `.player-primary-button`
in `app.css`. The bundled output confirmed `.player-primary-button` now
appears *before* `.player-icon-button`, so the main play/pause button
would render at 2.55rem with the translucent icon-button background
instead of 4.2rem with the brighter background.

**Resolution:** Moved `.player-primary-button` into `player.css`,
declared after `.player-icon-button`. Both classes always travel
together now, and the source order is explicit. Added a comment
explaining the equal-specificity ordering constraint.

## Finding #3 — MAJOR · ACCEPTED · FIXED (reverted)
**`.secondary-button` / `.danger-button` scoping is fragile**
(`Button.svelte:105-119`)

The reviewer is correct and the reasoning is sound. I framed these as
"Button-owned variants," but they are in fact plain-`<button>` affordances
used by 5 components that don't go through `<Button>`:
`HomeNavbar`, `PersonHero`, `TrailerPlayer`, `ProviderSettings`,
`SettingsShell`. Today it works only because all of those transitively
import `Button`, loading its CSS chunk. That's a load-bearing assumption
invisible at the call site — exactly the kind of latent footgun the
scoping refactor was supposed to *eliminate*, not introduce.

The reviewer offered three options; I took (a): keep them global. This
matches the existing `.icon-only` precedent (also a button affordance
any element can opt into) and the file's own stated invariant ("keeps
only: design tokens, global element rules, cross-component utilities").

**Resolution:** Reverted the `.secondary-button` / `.danger-button`
move — they're back in `app.css` (now using color tokens). Button.svelte
keeps only the truly Button-scoped `.is-busy` spinner.

## Finding #4 — MINOR · PARTIALLY ACCEPTED · FIXED
**`navRegion.ts` fallback path scrolls + `offsetParent` skips fixed elements**
(`navRegion.ts:126-139`, `firstFocusableIn:120-123`)

Two sub-issues, both valid:

1. **Scroll on initial focus:** The original "nothing focused" fallback
   called `focus()` without `scrollIntoView`. My `focusAndScroll` helper
   always scrolls. I agree this is likely an improvement (focusing an
   off-screen element without scrolling is usually a bug), but the
   reviewer is right that it should be a deliberate decision. **Decision:
   keep the scroll** — it's the correct behavior for a focus-management
   system, and the original omission looked like an oversight.

2. **`offsetParent !== null` filter:** My `firstFocusableIn` added a
   visibility filter via `offsetParent !== null`, which would skip
   `position: fixed` elements (they have `null` offsetParent even when
   visible). The original used bare `querySelector` with no filter.
   I checked: no current navRegion container holds fixed-position
   focusables (the fixed elements are `.page-backdrop`, `.player-overlay`,
   `.trailer-shell`, `.theme-song-*` — none are nav regions). But it's a
   latent risk if a modal ever registers as a region.

**Resolution:** Reverted `firstFocusableIn` to match the original — bare
`querySelector`, no visibility filter. The visibility concern is better
addressed at the `getVisibleFocusable()` layer (in spatialNavigation.ts),
not in the region entry-point helper.

## Finding #5 — NIT · ACCEPTED · FIXED
**`cards[0]` vs `cards.at(-1)` inconsistency**
(`HomeContent.svelte:138`)

Trivial but fair — mixing bracket access with `.at()` in the same
expression undercuts the `prefer-at` lint sweep's rationale.

**Resolution:** Changed `cards[0]` to `cards.at(0)`.
